# ConfigDumpInfo.xml

`ConfigDumpInfo.xml` содержит стабильные UUID объектов метаданных 1С.
Коннектор использует эти UUID при построении URN в DataHub. Благодаря этому
переименование объекта в 1С не создаёт в DataHub новый, не связанный со старым,
dataset.

## Режим подготовленного файла

Один из поддерживаемых сценариев — использовать заранее подготовленный файл.
Он удобен для разработки, первичной проверки и окружений, где ещё нет
автоматической выгрузки. Для продуктовой эксплуатации обычно лучше
автоматизировать получение `ConfigDumpInfo.xml` через exporter, CI/CD или
другой регламентный процесс, чтобы файл обновлялся вместе с конфигурацией 1С.

Ручной сценарий выглядит так:

1. Выгрузить `ConfigDumpInfo.xml` из Конфигуратора 1С или EDT.
2. Положить файл в каталог, доступный runner загрузки.
3. Указать `ONEC_CONFIG_DUMP_INFO_PATH`.
4. Выполнить предварительную проверку перед загрузкой.

```bash
python datahub/scripts/validate_configdump.py \
  --recipe examples/recipes/1c-full.dhub.yaml \
  --path /path/to/ConfigDumpInfo.xml \
  --check-scope
```

Проверка убеждается, что файл существует, корректно разбирается, содержит UUID
поддерживаемых объектов метаданных 1С и, при включённом `--check-scope`,
покрывает объекты, выбранные в recipe.

Если во время загрузки объект, табличная часть, HTTP/Web-сервис или endpoint
из выбранного scope отсутствует в `ConfigDumpInfo.xml`, загрузка завершается
ошибкой. Это защищает каталог от внешне успешного, но неполного обновления:
без UUID коннектор не может построить стабильный URN в DataHub. В такой
ситуации нужно обновить `ConfigDumpInfo.xml` из той же версии основной
конфигурации 1С и повторить загрузку.

## Проверка актуальности

Укажите `CONFIGDUMP_MAX_AGE_SECONDS`, чтобы загрузка падала, если файл слишком
старый:

```text
CONFIGDUMP_MAX_AGE_SECONDS=86400
```

Пустое значение, `0`, `off`, `false`, `none` и `skip` отключают проверку
возраста файла.

## Автоматическая выгрузка через exporter

`1c-metadata-service/configdump-exporter` содержит Docker-образ, который может
запускать режим Designer 1С и автоматически выгружать файл.
Использование контейнера зависит от доступности дистрибутивов платформы 1С и
возможности прокинуть лицензию в контейнер (варианты: прокинуть файл лицензии
с хоста или поднять сервер лицензирования, который может раздать лицензию).
Если лицензирование в целевом окружении не настроено, используйте
режим подготовленного файла.

Параметры подключения exporter не связаны с URL HTTP-сервиса метаданных.
Используйте один из вариантов:

```text
ONEC_SERVER=1c-host
ONEC_INFOBASE=1c-test
```

или:

```text
ONEC_IB_CONNECTION=Srvr="1c-host";Ref="1c-test";
```

или:

```text
ONEC_FILE_DB=/work/ib
```

`ONEC_INFOBASE` - это server Ref информационной базы для 1C Designer.
`ONEC_INFOBASE_NAME` - стабильное пространство имён DataHub, используемое в
URN и навигации. Эти значения могут отличаться.
