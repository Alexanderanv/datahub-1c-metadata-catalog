# Manual lineage в сервисе метаданных 1С

`GET /lineage` поддерживает ручные внутренние dataset-level связи 1С через
общий модуль `mds_ManualLineage`. Этот механизм нужен для потоков данных,
которые нельзя надежно вывести из метаданных платформы без статического анализа
BSL-кода.

## Поддерживаемые виды связей

Сейчас реализованы два разных механизма:

- `manual_dataset_flow` для `GET /lineage`: внутренняя dataset-level связь
  1С-объектов, которая маппится в стандартный DataHub `UpstreamLineage`.
- `integration_service_input` для `GET /integration-services`: metadata hint,
  что HTTP/Web endpoint внутри 1С использует конкретный 1С-объект. Это не
  полный process lineage и не `DataJobInputOutput`.

Семантика: downstream-объект использует данные upstream-объекта. DataHub
коннектор маппит такую связь в стандартный `UpstreamLineage` downstream
dataset-а. Направление всегда задается как `upstream -> downstream`.

`manual_process` и cross-platform manual lineage не реализованы в этом
контракте. Runtime-сервис метаданных 1С не отдаёт внешних потребителей
HTTP/Web endpoint-ов, а 1С DataHub-коннектор не должен эмитить `outputs` или
`DataJobInputOutput` для service jobs. Такие связи должен писать отдельный
integration source / overlay, который владеет полным набором input/output
edges для process node.

## Где задавать правила

Основная точка расширения:

```bsl
Процедура mds_ManualLineage.ДополнитьLineage(Связи, Контекст) Экспорт
```

`Связи` — массив, который нужно заполнить структурами ручных связей.
`Контекст` содержит служебные фильтры `filter_objects` и `filter_kinds`.
Обычно разработчику не нужно читать эти фильтры вручную: HTTP-сервис применит
их после получения правил.

Правила следует задавать через объекты метаданных, а не строковые литералы:

```bsl
Детали = mds_ManualLineage.НовыеДеталиСвязи(
    "mds_ManualLineage",
    "order_to_invoice"
);

mds_ManualLineage.ДобавитьСвязьНаборовДанных(
    Связи,
    Метаданные.Документы.ЗаказПокупателя,
    Метаданные.Документы.РасходнаяНакладная,
    "Расходная накладная использует данные заказа покупателя.",
    "high",
    Детали
);
```

Для внутренних inputs HTTP/Web endpoint-ов используется отдельная точка:

```bsl
Процедура mds_ManualLineage.ДополнитьИнтеграционныеСервисы(Связи, Контекст) Экспорт
    mds_ManualLineage.ДобавитьВходМетодаHTTPСервиса(
        Связи,
        "OrdersApi",
        "Orders",
        "Post",
        Метаданные.Документы.ЗаказПокупателя,
        "Метод публикует данные заказа покупателя."
    );
КонецПроцедуры
```

Такая связь попадёт в `IntegrationEndpoint.input_objects`, а DataHub connector
сохранит её в `DataJobInfo.customProperties` (`internalInputObjectFullNames`,
`internalInputDatasetUrns` при успешном резолве). `DataJobInputOutput` при этом
не эмитится. List-like значения в `customProperties` хранятся как JSON-массивы
в строке, потому что DataHub URN содержит запятые и не может безопасно
разбираться через comma-split.

## Callback в модуле менеджера

Если правило удобнее держать рядом с объектом, в общем модуле можно явно
вызвать экспортную процедуру менеджера:

```bsl
mds_ManualLineage.ВызватьДополнениеLineageМенеджера(
    Связи,
    Контекст,
    Метаданные.Документы.ЗаказПокупателя
);
```

В модуле менеджера соответствующего объекта нужно реализовать:

```bsl
Процедура mds_ДополнитьLineageОбъекта(Связи, Контекст) Экспорт
    mds_ManualLineage.ДобавитьСвязьНаборовДанных(
        Связи,
        Метаданные.Документы.ЗаказПокупателя,
        Метаданные.Документы.РасходнаяНакладная,
        "Описание ручной связи."
    );
КонецПроцедуры
```

HTTP-сервис не обходит все модули менеджеров автоматически. Вызов callback-а
должен быть явно добавлен в `mds_ManualLineage.ДополнитьLineage`, чтобы
контракт оставался предсказуемым и диагностируемым.

## Структура ручного правила

Конструктор `НовыеДеталиСвязи` возвращает структуру `Детали`:

| Поле | Обязательность | Значение |
| --- | --- | --- |
| `origin` | да | источник правила, например `mds_ManualLineage` или имя модуля менеджера |
| `rule_id` | нет | стабильный идентификатор правила |
| дополнительные поля | нет | JSON-совместимые значения из `ДополнительныеДетали` |

Конструктор `НоваяСвязьНаборовДанных` возвращает структуру связи:

| Поле | Обязательность | Значение |
| --- | --- | --- |
| `ВидСвязи` | да | `manual_dataset_flow` |
| `ОбъектИсточник` | да | объект метаданных upstream |
| `ОбъектПриемник` | да | объект метаданных downstream |
| `ИсточникСвязи` | да | `manual` |
| `Уверенность` | да | `high`, `medium` или `low` |
| `Описание` | нет | человекочитаемое описание |
| `Детали` | нет | структура/соответствие с JSON-совместимыми значениями |

Для совместимости с тестами и возможными служебными генераторами endpoint также
понимает англоязычные aliases `kind`, `upstream`, `downstream`, `source`,
`confidence`, `description`, `details`, но для 1С-кода предпочтительны
экспортные конструкторы общего модуля. Строковые aliases допустимы только для
служебного кода и также должны резолвиться в существующие объекты метаданных.

Для integration service inputs доступны конструкторы:

- `НоваяВнутренняяСвязьИнтеграционногоСервиса(ПолноеИмяЭндпоинта, ОбъектИсточник, ...)`
- `ДобавитьВходМетодаHTTPСервиса(Связи, ИмяСервиса, ИмяШаблонаURL, ИмяМетода, ОбъектИсточник, ...)`
- `ДобавитьВходОперацииWebСервиса(Связи, ИмяСервиса, ИмяОперации, ОбъектИсточник, ...)`

Endpoint задаётся canonical full name:

- `HTTPService.<service>.URLTemplate.<template>.Method.<method>`
- `WebService.<service>.Operation.<operation>`

Структура `integration_service_input`:

| Поле | Обязательность | Значение |
| --- | --- | --- |
| `ВидСвязи` | да | `integration_service_input` |
| `Эндпоинт` | да | canonical full name HTTP method / Web operation |
| `ОбъектИсточник` | да | 1С-объект, используемый endpoint-ом |
| `ИсточникСвязи` | да | `manual` |
| `Уверенность` | да | `high`, `medium` или `low` |
| `Описание` | нет | человекочитаемое описание |
| `Детали` | нет | структура/соответствие с JSON-совместимыми значениями |

## Диагностика

Ошибки manual lineage не проглатываются. Если правило содержит неподдерживаемый
`ВидСвязи`, пустой upstream/downstream или объект, для которого невозможно
получить полное имя метаданных, `GET /lineage` завершится ошибкой. Это
сознательное fail-fast поведение: некорректное ручное правило не должно тихо
создавать неполный lineage-граф.

Фильтр `objects` в `/lineage` применяется после получения ручных правил.
Если upstream или downstream не входят в текущий scope ingestion, связь не
будет возвращена и не попадет в DataHub.

Фильтры `services` и `endpoints` в `/integration-services` применяются к
ручным `integration_service_input` так же строго: связь не попадёт в ответ,
если её endpoint или service не входят в текущий recipe scope.
